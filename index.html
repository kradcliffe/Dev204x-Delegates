Basic Delegates

Credit to Dan Wahlin for excellent explanations.
Any and all mistakes are my (kradcliffe’s) responsibility.

What’s the point?
Going through this and a couple more exercises will help you much more deeply understand what Linq ACTUALLY does.

Delegates are method pointers, plain and simple.
You use the delgate keyword, with a named method signature
This allows you to point the delegate at any regular method with the same signature.
(That is: Having identical return type and method parameters, but with different name)

Exercises

You can see above that at the lowest level, the way you connect delegates to actual methods is to instantiate them (new).
When you do this, you pass in the method you wish to invoke THROUGH the delegate.

After you have set this up, notice that you can then invoke the delegate directly, as though it were a normal method.

Use your own editor (VS recommended) or fork this fiddle:
DotNetFiddle

And perform the following exercises:

  1. Try to create another method with the same signature.(Same return type, and params as the delegate, different name)
    Similar to what exists above for:

void WorkPerformed1(int hours, WorkType workType)

Create a second instance of the delegate and new it up, pointing to your new method.
Mentally note what the constructor is expecting when you open parentheses.

  1. Alter your methods to do something with the input parameters.
    Make sure that your output shows the parameters some how.

  2. Alter your delegate to return an int.
    Play with the code (your methods need to be altered) to make sure it compiles, and will return a result

  3. Create a second delegate with a different signature. (different return type and/or different params)
    Create another method (with the appropriate signature), and point your delegate to your new method.

  4. (Challenge) Point your delegate to an anonymous method.
    This will be covered in more detail in another assignment, so don’t worry if this doesn’t make sense.
